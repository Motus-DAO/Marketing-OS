# Notion Routing Rules

## Core Principle
Notion is the operational source of truth. Workspace files define process and standards, but status and execution state should be reflected in Notion.

## Main Databases and Pages

### Ideas
Use for:
- raw or refined ideas
- campaign angles
- content opportunities
- product/growth hypotheses

Do not use for:
- final copy
- execution checklists
- published asset tracking

### Content Engine
Use for:
- approved content assets
- hooks
- core idea
- format
- platform
- CTA
- draft copy
- production notes

Do not use for:
- raw brainstorming dumps
- unrelated project tasks

### Tasks
Use for:
- execution work
- writing tasks
- design tasks
- publishing tasks
- QA tasks
- review tasks

Each task should represent one operational action.

### Projects
Use for:
- broader initiatives
- campaign-level work
- strategic efforts
- recurring operating tracks

### Metrics
Use for:
- tracking campaign and funnel performance
- key KPI snapshots
- ongoing performance review

## Flow Rules

### Idea to Content
1. Idea is selected in Ideas.
2. Approved asset is created in Content Engine.
3. Execution work is created in Tasks.

### Content to Tasks
Every meaningful content asset should create tasks for:
- writing or refinement
- design or production
- QA or approval
- posting or publishing

### Campaign Tracking
If multiple assets belong to a campaign, tie them back to a Project where possible.

## Routing by Work Type
- Research notes -> Ideas or dedicated research pages
- Copy drafts -> Content Engine
- Design briefs -> Content Engine notes or brand pages
- Execution work -> Tasks
- Weekly reporting -> Metrics or project notes

## Status Ownership
- Idea maturity lives in Ideas
- Asset maturity lives in Content Engine
- Execution state lives in Tasks
- Initiative state lives in Projects
- Outcome tracking lives in Metrics
