# Sunrise OS

## Origin note
This document captures the origin logic of MotusDAO OS as discussed during the initial architecture phase.

## Core intent
The goal is not just to manage content or tasks. The goal is to build a reusable operating system that can turn complex initiatives into structured, repeatable narrative systems.

This system should help:
- reduce conceptual complexity
- turn ideas into offers
- turn offers into funnels
- turn funnels into revenue
- capture what works
- replicate it across projects

## Human-facing vs machine-facing split
Recommended architecture:
- Notion = human workspace, planning, review, publishing, collaboration
- Convex = machine workspace, structured state, compressed intelligence, automation runtime
- Project files = standards, prompts, schemas, agent rules

## Business objective
Create steady revenue streams from the selected channels by understanding:
- algorithms
- audience behavior
- market response
- conversion patterns
- monetization pathways
- repeatable narratives
- asset performance

This should apply across:
- MotusDAO
- Avril
- Brahma101.eth
- future projects

## Strategic challenge
These initiatives contain too many concepts and too much narrative complexity to manage well through long conversations and raw context alone.

The OS must:
- keep structured logs of what works
- make context retrieval efficient
- avoid prompt stuffing
- reduce hallucinations
- support replication into future projects

## Memory philosophy
The memory model should follow:
- working memory
- long-term memory
- compressed context

We should store:
- summary
- decisions
- next actions
- references

We should avoid storing raw long conversations as the main system of memory.

## Core design principle
Store less, structure more, summarize continuously.

This is intended to keep the system efficient, reduce token waste, and make future agent execution more reliable.

## Project-first model
Every record in Convex should belong to a project first.

Initial projects:
- MotusDAO
- Avril
- Brahma101.eth

That makes replication possible and keeps each narrative system isolated while reusing the same operating pattern.

## What Convex should own
Convex should hold:
- runtime state
- operational memory
- performance intelligence
- project-scoped context
- compressed summaries
- learnings
- decisions
- references
- automation and agent state

## Why this matters
This architecture is designed so that MotusDAO is the first implementation, then the same system can propagate, replicate, and escalate across more projects.

## Key line
The OS must convert complex initiative clusters into structured, repeatable narrative systems.

## Expansion principle
We start with MotusDAO, then propagate, replicate, and escalate.
